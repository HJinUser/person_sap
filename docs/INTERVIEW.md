# 면접 예상 질문 & 답변 정리

포트폴리오 설명 시 실제로 나올 법한 질문과, 이 프로젝트의 코드로 뒷받침되는 답변.

---

## SAP 관련

**Q. SAP을 실제로 써본 적이 없는데 이 프로젝트가 무슨 의미가 있나요?**

신입에게 SAP 운영 경험을 기대하기 어렵다는 건 알고 있습니다. 대신 저는 SAP의
공개 문서로 HCM 모듈의 데이터 모델(인포타입 PA0000/PA0001/PA0008/PA2001)과
연동 표준(NetWeaver Gateway OData v2)을 학습해서 직접 재현했습니다.
`$filter`·`$inlinecount` 같은 쿼리 옵션, `{"d": {"results": []}}` 응답 구조,
X-CSRF-Token 핸드셰이크까지 실제 규격대로 구현했기 때문에, 실무에서 진짜 SAP
Gateway를 만나도 소비하는 쪽 코드는 거의 그대로 쓸 수 있습니다.

**Q. 인포타입(Infotype)이 뭔가요?**

SAP HCM에서 인사 데이터를 주제별로 묶은 논리 테이블 단위입니다. PA0001은
조직배치(부서·직위), PA0008은 기본급, PA2001은 결근/휴가. 특징은 **기간 유효
(BEGDA~ENDDA) 레코드** 방식이라는 것 — 급여가 바뀌면 기존 레코드를 수정하지
않고 기간을 닫고 새 레코드를 만듭니다. 제 프로젝트의 salary_history가 그
구조를 그대로 따르고, 덕분에 "1년 전 급여" 같은 시점 조회가 가능합니다.

**Q. OData v2와 v4의 차이는?**

v4가 더 최신(JSON 기본, $expand 개선 등)이지만 SAP NetWeaver Gateway 실무
표준은 여전히 v2입니다. 그래서 v2 규격($format, __metadata, d-wrapper)으로
구현했습니다.

**Q. 실제 SAP 시스템과 연동한다면 뭐가 달라지나요?**

엔드포인트 주소와 인증(Basic/OAuth)만 바뀌고 소비 코드는 동일합니다. 추가로
필요한 것: 변경 요청 시 CSRF 토큰 실사용, 대량 조회 시 $batch, 증분 동기화용
델타 쿼리, 그리고 실데이터의 결측·오입력 처리 로직입니다.

## ML 관련

**Q. 왜 정확도(accuracy)가 아니라 ROC-AUC로 평가했나요?**

퇴사율이 14%라 전원을 '재직'으로 찍어도 정확도 86%가 나옵니다. 불균형
데이터에서는 순위 판별력을 보는 ROC-AUC와 소수 클래스 관점의
정밀도/재현율/F1이 맞습니다. 추가로 `class_weight='balanced'`와 임계값
튜닝(0.40)으로 재현율을 끌어올렸습니다.

**Q. 데이터 누수(leakage)를 어떻게 막았나요?** ⭐ 이 프로젝트의 핵심

두 가지를 처리했습니다.
① **시점 기준 피처**: 퇴사자의 피처를 "지금" 기준으로 계산하면 퇴사 이후
정보가 섞입니다. 퇴사자는 퇴사일, 재직자는 관측일 기준으로 피처를 계산했습니다.
② **관측 시점 편향**: 처음에 재직자 전원을 같은 날짜(6/30) 기준으로 계산했더니
AUC가 0.98이 나왔습니다. 원인을 추적해 보니 3월 일괄 인상 직후라는 시점 특성이
재직자에게만 공통으로 생겨서, 모델이 '마지막 인상 후 경과시간 ≈ 3개월이면
재직자'라는 **관측 시점의 흔적**을 학습한 거였습니다. 재직자 기준일을 최근
1년 내 무작위로 샘플링해 제거했고 AUC는 0.89로 내려왔습니다 — 성능이 떨어진 게
아니라 부풀려진 성능을 바로잡은 겁니다.

**Q. AUC 0.89면 좋은 건가요?**

합성 데이터라 실전보다 낙관적인 수치입니다. 다만 이 프로젝트의 검증 목표는
절대 성능이 아니라 "데이터에 심어둔 인과 요인(잔업·병가·급여 정체·평가 하락)을
모델이 복원하는가"였고, 순열 중요도 상위권이 정확히 그 요인들로 나왔습니다.

**Q. 피처 중요도는 어떻게 구했나요?**

순열 중요도(permutation importance)입니다. 특정 피처를 무작위로 섞었을 때
테스트 AUC가 얼마나 떨어지는지 측정합니다. 트리 모델 내장 중요도와 달리 모델
종류와 무관하게 같은 방식으로 계산돼서, 최적 모델이 LR→RF→GB 무엇으로 바뀌어도
해석 파이프라인이 유지됩니다.

**Q. 고위험 판정 기준은?**

절대 확률이 아니라 **재직자 위험도 분포의 상위 5%** 입니다. 랜덤 포레스트 등
트리 앙상블의 확률 출력은 보정(calibration)되지 않아 절대값의 의미가 약하고,
실무 용도가 '우선 면담 대상 선정'이므로 우선순위(랭킹) 방식이 목적에 맞습니다.

## 아키텍처 관련

**Q. 왜 ML을 Java(Spring)가 아니라 별도 Python 서비스로 뒀나요?**

ML 생태계(scikit-learn, pandas)가 Python에 있고, ERP 쪽은 Java가 표준이라
언어 경계가 불가피합니다. 이걸 REST API 계약으로 풀었는데, 실제 SAP BTP의
side-by-side 확장 패턴과 같은 구조입니다 — ERP 코어를 수정하지 않고 표준
API로만 통신하는 것. 배포·스케일링도 서비스별로 독립적입니다.

**Q. AI 리포트에서 환각(hallucination)은 어떻게 막나요?**

LLM에게 원본 데이터를 주지 않습니다. 규칙 기반 엔진이 먼저 검증된 통계
사실(부서별 수치, 개인별 태그)을 JSON으로 추출하고, LLM은 그것을 자연어로
서술하는 역할만 합니다. 프롬프트에 "JSON에 있는 값만 사용" 제약을 명시했고,
API 키가 없거나 호출이 실패하면 규칙 기반 리포트로 자동 대체됩니다.

**Q. 이 시스템을 운영에 올린다면 뭘 추가해야 하나요?**

인증/인가(대시보드는 인사팀만), 학습 이력 관리(모델 버전·데이터 스냅샷),
스케줄러(주기적 재학습), 예측 드리프트 모니터링, 그리고 개인정보 처리
관점의 접근 로그와 가명화입니다.

## 윤리 관련

**Q. 퇴사 예측이 직원 감시 아닌가요?**

민감한 주제라는 점을 인지하고 설계에 반영했습니다. ① 용도를 '해고 판단'이
아니라 '조직 개선 신호'(면담, 보상 검토, 업무 부하 조정)로 한정하고 대시보드와
리포트에 명시했습니다. ② 예측 근거(위험 요인 태그)를 투명하게 제공해 블랙박스
판단을 피했습니다. ③ 실제 도입이라면 노사 합의, 개인정보 영향평가가 선행되어야
한다고 생각합니다.
